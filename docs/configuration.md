# INVARIANT Configuration Reference

The single source of truth for project-level settings is **`invariant.config.json`** in the project root. It is generated by `npx @santh/invariant init`, read by the agent and edge sensor, and is safe to commit (no secrets ŌĆö use environment variables or platform secrets for keys).

---

## Config file location

- **Default**: `{process.cwd()}/invariant.config.json`
- **Override**: Middleware and CLI accept `configPath` to point to a different file.

## Modes

### Config-level modes (`invariant.config.json` ŌåÆ `mode`)

| Mode | Meaning |
|------|--------|
| **monitor** | Detect and log only; never block. Safe for rollout and tuning. |
| **enforce** | Middleware may block requests when detections exceed thresholds (see middleware mode mapping below). |
| **off** | Detection can be disabled (implementation may still run in no-op or minimal form). |

### Middleware-level modes (Next.js, Hono, Fastify, Koa)

Middleware that supports extended modes accepts `mode` in options. It maps config `mode: 'enforce'` to a default of **defend** when no explicit `mode` is passed.

| Mode | Behavior |
|------|----------|
| **monitor** | Run detection; call `onDetect`; **never block**. |
| **sanitize** | Strip dangerous patterns from body/query (e.g. path traversal, template literals); **block only on critical** severity. |
| **defend** | **Block** when any match has **critical** or **high** severity and confidence exceeds the class threshold. |
| **lockdown** | **Block** on any detection (any severity above threshold). |

**Recommendation**: Start with `monitor`, add exception rules for known-good traffic, then switch to `defend`. Use `lockdown` only in high-security segments.

## Exception rule syntax

Exception rules let you skip detection or blocking for specific requests. Used by Next.js, Hono, Fastify, and Koa middleware as `exceptionRules: ExceptionRule[]`.

Each rule is an object; **all specified fields must match** for the rule to apply. Omitted fields are not tested.

| Field | Type | Description |
|-------|------|-------------|
| **path** | string \| RegExp | Request path (e.g. `'/api/health'` or `/^\/webhooks\/.*/`). |
| **method** | string \| string[] | HTTP method(s), e.g. `'GET'` or `['GET','HEAD']`. |
| **ip** | string \| RegExp | Client IP from the **connection** (do not use `X-Forwarded-For` for security decisions). |
| **surface** | string | Limit exception to this surface: `query_param` \| `body_param` \| `header` \| `cookie` \| `path` \| `ip`. |
| **key** | string \| RegExp | Parameter or key name (e.g. `'callback'` or `/^x-custom-/`). |
| **class** | string \| string[] | Invariant class ID(s) to exempt (e.g. `'graphql_introspection'` or `['sql_time_oracle','sql_error_oracle']`). |

**Examples:**

```json
{ "path": "/api/health", "method": "GET" }
```
Skips all detection for `GET /api/health`.

```json
{ "path": "/webhooks/stripe", "method": "POST" }
```
Skips detection for Stripe webhook POSTs.

```json
{ "surface": "header", "key": "x-request-id" }
```
Skips detection only for the header `x-request-id` (other surfaces still checked).

```json
{ "class": "graphql_introspection" }
```
Never blocks on `graphql_introspection` alone (other classes still apply).

**Security**: IP-based rules must use the **verified connection IP** (e.g. `request.ip` set by the framework). Do not use `X-Forwarded-For` or `X-Real-IP` for allowlisting ŌĆö they are attacker-controlled.

## Signals

Optional `signals` object for central signal ingest:

| Key | Type | Description |
|-----|------|--------------|
| **ingestUrl** | string | URL for signal upload (e.g. Santh central ingest). |
| **batchSize** | number | Number of signals to buffer before flush. |
| **sensorToken** | string | Auth token for ingest. **Do not commit** ŌĆö use env or secrets. |

Example (with token from env):

```json
{
  "v": 1,
  "category": "saas",
  "mode": "monitor",
  "signals": {
    "ingestUrl": "https://ingest.santh.io/v2",
    "batchSize": 100
  }
}
```

Set `sensorToken` via environment or platform secrets, not in the file.

## Thresholds

Optional **`thresholds`** override the engineŌĆÖs per-severity block thresholds. Values are **confidence ceilings**: a match with confidence **Ōēź** the threshold for its severity can trigger a block (when mode is enforce/defend).

| Key | Type | Engine default (approx.) | Description |
|-----|------|---------------------------|-------------|
| **critical** | number | 0.45 | Block when critical-class confidence Ōēź this. |
| **high** | number | 0.65 | Block when high-class confidence Ōēź this. |
| **medium** | number | 0.80 | Block when medium-class confidence Ōēź this. |
| **low** | number | 0.92 | Block when low-class confidence Ōēź this. |

Example:

```json
{
  "v": 1,
  "category": "api",
  "mode": "enforce",
  "thresholds": {
    "critical": 0.4,
    "high": 0.6,
    "medium": 0.85
  }
}
```

Lower values = **stricter** (block more). Raise values to reduce false positives. Threshold tuning should be done in **monitor** mode first, then switch to **enforce** or **defend**.

---

## Threshold tuning guide

1. **Start in monitor**  
   Run with `mode: 'monitor'` and collect logs/callbacks. No blocking.

2. **Add exception rules**  
   For paths/methods/surfaces that legitimately trigger detections (e.g. security scanners, webhooks, admin tools), add `exceptionRules` so they are not blocked when you enable enforce/defend.

3. **Optional: relax thresholds**  
   If critical/high classes fire on benign input, **increase** that severityŌĆÖs threshold (e.g. `high: 0.75`) so only stronger signals block. Do not lower thresholds below engine defaults unless you have a clear reason (e.g. internal pentest).

4. **Optional: tighten thresholds**  
   To block earlier on critical classes, **decrease** the value (e.g. `critical: 0.35`). Use with care to avoid blocking legitimate traffic.

5. **Switch to enforce/defend**  
   Set config `mode: 'enforce'` and/or middleware `mode: 'defend'`. Verify blocks only occur for real attacks or expected test payloads.

6. **Use lockdown sparingly**  
   `lockdown` blocks on any detection. Use only in high-sensitivity environments after thorough exception and threshold tuning.

## Plugins

**`plugins`**: array of strings (paths to plugin modules).

Plugins add or replace invariant class modules at runtime. Each plugin must export an object satisfying `InvariantPlugin`: `name`, `version`, and `classes` (array of `InvariantClassModule`). See [API reference ŌĆö Plugin system](./api-reference.md#plugin-system).

Example:

```json
{
  "v": 1,
  "category": "saas",
  "mode": "monitor",
  "plugins": ["./plugins/custom-classes.js", "@santh/plugin-allowlist"]
}
```

Paths are resolved relative to the config file or process cwd depending on the loader.

## Validation

Config is validated with `validateConfig(raw)` (from `@santh/invariant` engine). Invalid or unknown fields are ignored; invalid values for known fields throw `ConfigError` with a message. Defaults from `DEFAULT_CONFIG` are applied for missing top-level keys.
